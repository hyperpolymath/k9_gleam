== Claims Substantiation

=== Claim 1: "Full K9 parser with validation and error recovery"

**How it works:** K9 is a data serialization format (similar to YAML/TOML) with named-by-default semantics, type inference, and first-class support for comments and metadata. The Gleam parser (`src/k9/parser.gleam`) implements recursive descent parsing: recognizes key-value pairs, lists, maps, type annotations, and metadata directives. Each parse rule returns `Result(Parsed, ParseError)` with position tracking, enabling error recovery and helpful error messages. The parser validates K9 syntax (balanced braces, valid types, no duplicate keys) without requiring an external schema. Type inference in `src/k9/types.gleam` assigns inferred types to values (e.g., `42` is Nat, `"hello"` is String, `[1, 2, 3]` is List(Nat)).

**Caveat:** K9 is a younger format than YAML/TOML, so ecosystem tooling is sparse. The Gleam parser is reference implementation, but integration with existing tools requires translation layers. Schema validation is external (not built into the parser); for strictness, pair with a schema validator.

**Evidence:** `src/k9/parser.gleam` implements `parse(text: String) -> Result(Document, ParseError)`. `src/k9/types.gleam` defines K9 type system. Tests in `test/` verify parsing on canonical K9 documents (from https://github.com/hyperpolymath/k9-spec).

=== Claim 2: "Cross-compilation to BEAM and JavaScript via Gleam's dual-target support"

**How it works:** Gleam source code compiles to either Erlang bytecode (BEAM) or JavaScript (Node.js, browser, Deno) using the same `.gleam` source files. The k9_gleam library has no platform-specific code—it uses only Gleam stdlib functions that abstract over BEAM/JS differences. When building the library, Gleam detects both targets and generates both sets of compiled output. Users import k9_gleam in Elixir/Erlang projects (BEAM target) or in JavaScript-based Gleam projects (JS target), and get the same parser logic compiled to their target runtime. Hex.pm (Erlang registry) and Gleam's package manager both distribute the library for both targets.

**Caveat:** Performance differs: BEAM version benefits from Erlang's lightweight processes and hot-code reloading, while JavaScript version is faster for simple parsing tasks (less overhead). Some edge cases (error recovery on unterminated strings) may behave differently across runtimes due to stdlib differences. The library must avoid platform-specific idioms.

**Evidence:** `gleam.toml` specifies no platform conditionals. `src/k9/parser.gleam` uses only Gleam stdlib. CI tests both `gleam build` (BEAM) and `gleam build --target javascript`. Hex.pm publishes both artifacts.
